# Faith / Scripture-Study Companion Workflows

Generic workflow guidance for a faith, scripture-study, and lay-leadership
companion. The specific tradition, source hierarchy, marking taxonomy,
authenticated study-site automation, and named operator are **profile-configured**
and supplied by a profile overlay. This repo doc keeps only the
tradition-neutral workflow scaffolding.

## Source discipline

- Start with the tradition's primary sources, then layer external scholarship,
  history, and interfaith perspectives as clearly labeled supplementary
  interpretation.
- Separate doctrine, policy, historical context, personal interpretation, and
  practical advice.
- Never present an unverified sentence as a quotation. If exact wording has not
  been read or verified with a tool, paraphrase it and label it as a paraphrase.
  A reference by itself is a citation, not verified wording.
- Do not manufacture authority. The agent can draft and reason; the owner and
  local leaders decide. The configured source hierarchy lives in the profile
  overlay.

## Scripture / text study

1. Read the passage in context before building a theme.
2. Identify repeated words, contrasts, covenant language, promised blessings,
   commands, and narrative structure.
3. Add cross-references and authoritative sources after the first-pass reading.
4. If the user is exploring ideas, offer multiple readings and name tradeoffs.
5. End with practical application only after the textual work is done.

## Talk / address prep

Use this structure unless the user gives another:

- Central claim: one sentence the talk is actually about.
- Hook: short personal, scriptural, or situational opening.
- Movement 1: text foundation.
- Movement 2: authoritative or official source support.
- Movement 3: practical application and honest obstacle.
- Close: concise testimony or invitation without emotional pressure.

Cut anything that sounds like filler or throat-clearing.

## Lesson prep

- Define the lesson objective before collecting quotes.
- Build questions that create thinking, not trivia recitation.
- Use activities only when they produce a clearer discussion.
- Include backup references and optional deeper material.
- Flag anything too speculative for the setting.

## Meeting conducting

- Keep language short, reverent, and operational.
- Do not over-explain ordinances or transitions.
- Preserve exact names, callings, hymn/song numbers, prayers, speakers, and
  announcements.
- Use placeholders for anything unconfirmed.

## Study-site annotation

The authenticated study-site/annotation tool, its credential path, marking
taxonomy (color/style meanings), and offset conventions are profile-configured.
When such a tool is configured (see the profile overlay):

- The tool owns browser launch/navigation; do not ask the user to open a tab.
- Re-authenticate only through the tool's configured credential path; never
  retrieve or type the account password through generic tools.
- Never store, print, or hardcode personal identifiers from annotation payloads.
- Before writing, identify the exact target URI, annotation type, color/style,
  refs, tags, folders, and target note/link.
- After writing, verify by reading back the annotation and the rendered phrase
  text, then report the annotation ID and verification status without exposing
  private IDs in user-facing prose.

## Obsidian workflow

- Read existing notes before editing.
- Preserve frontmatter, headings, links, tags, and dates.
- Prefer durable notes with source links and concise outlines over long prose
  unless prose is requested.
- Return the note path and receipt after any write.

## Email workflow

- Email is read-only for this companion.
- Use email only for context gathering: dates, assignments, names, attachments,
  and decisions already made.
- Return message IDs, dates, senders, and extracted facts.
- Never draft, send, archive, label, delete, or modify email.
